# 口頭報告講稿：The Setwise Paradigm

> **主題：** Setwise 排序範式 — 在 LLM 零樣本文件排序中平衡效果與計算效率  
> **目標聽眾：** 資訊檢索課程同學與教授  
> **預估時長：** 約 10–12 分鐘  

---

## 一、開場問候

大家好，今天我要跟大家介紹的是一個在 LLM 零樣本文件排序領域中非常重要的研究方向——**Setwise Paradigm**。

我們都知道，隨著大型語言模型的興起，用 LLM 來做文件排序已經不是什麼新鮮事了。但問題是：**我們真的能同時兼顧排序品質和計算效率嗎？** 現有的方法似乎都在告訴我們——不行。今天這篇研究，就是要挑戰這個限制。

---

## 二、核心重點說明

### 2.1 現有方法的瓶頸：速度 vs. 品質的兩難

首先，讓我們來看目前 LLM 排序的三大經典範式，它們各有各的問題：

- **Pointwise（逐點式）**：每個文件獨立評分，速度很快，但它完全缺乏跨文件的比較能力，所以排序品質是最差的。
- **Pairwise（成對式）**：兩兩文件做比較，精確度非常高，但它的複雜度是 $O(N^2 - N)$，文件一多就直接爆炸，完全無法 scale。
- **Listwise（列表式）**：把一組文件丟進 LLM，讓它生成一個排序結果。效果居中，但需要大量的 token 生成，計算成本也不低。

如果我們把「排序品質」放在 Y 軸、「計算效率」放在 X 軸，這三個方法分布在不同的象限，但**右上角那個象限——既高品質又高效率——是空的**。這就是所謂的 "Missing Quadrant"，也是這篇研究要攻克的目標。

### 2.2 架構層面的限制

從更技術的角度來看，這些方法的限制可以整理成一張表：

| 方法 | 前向傳播次數 | 使用快速 Logits | 支援 Batching | 支援 Early Stopping |
|------|-------------|----------------|--------------|-------------------|
| Pointwise | $O(N)$ | ✅ | ✅ | ❌ |
| Listwise Generation | $O(r \cdot N/s)$ | ❌ | 部分 | ✅ |
| Pairwise AllPair | $O(N^2 - N)$ | ❌ | ✅ | ❌ |
| Pairwise Heapsort | $O(k \cdot \log_2 N)$ | ❌ | ❌ | ✅ |

這裡有一個關鍵觀察：**Pointwise 是唯一能用 logit extraction 的方法**。Logit extraction 只需要一次前向傳播就能拿到機率分布，不需要一個 token 一個 token 地生成，所以速度極快。相比之下，Listwise 和 Pairwise 都依賴 token generation，每多生成一個 token 就多一次推論，成本線性增長。

### 2.3 Setwise 的核心創新：三合一的混合引擎

那 Setwise 怎麼解決這個問題？它的設計非常巧妙，**同時融合了三個範式的優點**：

1. **來自 Pairwise 的高效排序與 Early Stopping**：不需要比完所有文件，只要找到 top-k 就可以提前結束。
2. **來自 Listwise 的多文件上下文感知**：一次看多份文件，讓模型在比較時有全局視野。
3. **來自 Pointwise 的快速 Logit 提取**：用單次前向傳播直接拿到排序信號，省去昂貴的 token 生成。

### 2.4 Setwise Prompt 的運作方式

具體來說，Setwise 的 prompt 長這樣：

> 給定一個查詢 {query}，以下哪個段落與查詢最相關？  
> Passage A: {passage_1}  
> Passage B: {passage_2}  
> Passage C: {passage_3}  
> ...  
> 請只輸出最相關段落的標籤。

關鍵在於：模型不需要真的「生成」答案。我們只需要看它在 A、B、C 等標籤上的 **logit 分數**，就能一次性得到所有候選文件的相對排序。例如 B: 0.51、A: 0.32、C: 0.11——這樣一次前向傳播就完成了多文件的偏好排序。

這個設計的精妙之處在於：它在一次計算中就提取了 **multi-way preference**，而不是像 Pairwise 那樣需要兩兩比對，也不像 Listwise 那樣需要生成完整的排序序列。

### 2.5 演算法壓縮：升級排序樹

接下來我們來看 Setwise 在演算法層面帶來了什麼改變。

傳統 Pairwise 使用二元 heap 來做排序，每個節點只跟兩個子節點比較。以 9 個節點為例，Pairwise Heapify 至少需要 **6 次比較**才能完成。但 Setwise 把比較集合從 2 擴展到 $c$，也就是每次同時比較 $c$ 個候選文件。當 $c=4$ 時，同樣 9 個節點只需要 **2 次比較**就能完成 heapify——這是一個非常大的壓縮。

直觀來說，Setwise 把排序樹的深度壓扁了，因為每一層能處理的節點數大幅增加，所以 LLM 推論的總次數也跟著大幅下降。

### 2.6 加速線性迭代

同樣的思路也適用於 Bubble Sort。傳統 Pairwise Bubble Sort 每次只比較相鄰的 2 個元素，複雜度是 $O(k \cdot N)$。而 Setwise Bubble Sort 每次比較一個包含 $c$ 個元素的窗口，只需要把最大的元素「冒」上去，複雜度直接降到 $O(k \cdot N/(c-1))$。

舉個例子，如果 $c=4$，那每一步就能涵蓋 4 個元素的比較，步數直接砍成原來的三分之一。這對實際的查詢延遲影響非常明顯。

### 2.7 拯救 Listwise：透過 Likelihood Estimation

Setwise 不只能搭配排序演算法，它甚至可以用來**改造 Listwise 本身**。

傳統的 Listwise Generation 需要 LLM 一個 token 一個 token 地生成排序結果，例如先輸出 "Passage 2"，再輸出 "Passage 1"，再輸出 "Passage 3"——延遲很高、而且還有格式錯誤的風險。

Setwise 的做法是：用同樣的 Listwise prompt，但不做 token generation，改為直接提取每個文件標籤的 **logit likelihood**。例如 P2 的 logit 是 0.6、P1 是 0.3、P3 是 0.1，那排序直接就是 [2] > [1] > [3]。**一次前向傳播、零延遲、零格式錯誤。**

這個方法把 Listwise 從一個「慢而準」的方法，變成了一個「又快又準」的方法。

## 三、實驗驗證與結果

### 3.1 命中目標：實證結果

講了這麼多理論，數據到底有沒有支撐？答案是完全有。

在 TREC DL 2019 和 TREC DL 2020 這兩個標準評測資料集上，Setwise.heapsort 相比 Pairwise 方法，**計算成本減少了 62%**，而 NDCG@10 的差異僅有 **0.8%**——幾乎可以忽略不計。

從散點圖可以清楚看到：Setwise 的各種配置（標記為黃色和橘色的點）穩穩地佔據了**左上角的最佳區域**——也就是低延遲、高品質。而 Pairwise（紅色和棕色的點）雖然品質接近，但延遲動輒 30 到 60 秒以上。Pointwise（藍色的點）速度快但品質明顯落後。

### 3.2 對初始排序的免疫力

還有一個很重要的特性：**Setwise 對第一階段檢索器的排序品質幾乎完全免疫**。

實驗中測試了三種初始排序：正常的 BM25、倒序的 BM25、以及隨機排序的 BM25。結果 Listwise Generation 的 NDCG@10 從大約 0.8 直接崩跌到 0.1 左右——對初始排序極度敏感。但 Setwise 的表現幾乎是一條水平線，不管初始排序多糟糕，最終的排序品質都穩定在 0.8 左右。

這是因為 Setwise 把文件當作一個**無序集合**來處理，不受輸入順序的影響。這在實際部署中是一個巨大的優勢——你不需要擔心第一階段檢索器的品質好不好。

### 3.3 跨架構通用、經濟可擴展

最後，Setwise 不挑模型。實驗在三類不同的 LLM 上都驗證了它的效果：

- **Flan-T5**：Setwise 把查詢延遲壓到約 8 秒，全面超越所有 Pairwise 和 Listwise 的配置。
- **LLaMA-2 & Vicuna**：維持最優的 NDCG@10 分數，同時完全避開了自迴歸生成的瓶頸。
- **GPT-3.5 Commercial API**：Listwise 每個查詢的 API 成本是 \$0.045，而 Setwise 只要 **\$0.029**——**成本直接砍掉將近一半**，效果卻完全一樣。

這代表無論你用開源模型還是商業 API，Setwise 都能帶來實質的效率提升。

## 四、結語與下一步

總結一下，今天的三個核心 takeaway：

1. **停止生成 token。** 用 Setwise logit extraction 繞過自迴歸瓶頸，一次前向傳播就拿到排序信號。
2. **壓扁你的排序樹。** 同時比較 $c$ 個文件，從數學上大幅削減 LLM 推論次數。
3. **通用部署。** 達到 Pairwise 等級的排序品質、Pointwise 等級的速度，在開源模型和商業 API 上都穩定有效。

Setwise 範式成功填補了 LLM 排序中那個「高品質 + 高效率」的空白象限。它不是某一個範式的改良，而是從根本上重新定義了我們該怎麼用 LLM 做文件排序。

對我們來說，這個研究的啟示是：**在設計 LLM-based 的檢索系統時，prompt 的結構設計本身就是一種架構決策**。選擇讓模型「生成答案」還是「表達偏好」，會從根本上影響系統的效率天花板。

未來可以進一步探討的方向包括：

- Setwise 在更大規模 LLM（如 GPT-4、Gemini）上的效果驗證
- 與傳統 dense retrieval 或 learned sparse retrieval 的結合方式
- 在 RAG 系統中作為 reranker 的端到端效能評估
- 動態調整候選集合大小 $c$ 的自適應策略

以上就是我的報告，謝謝大家。
